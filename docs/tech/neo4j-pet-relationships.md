# 宠物关系定义（Player / Pet / Bond）

本文件聚焦两类“会随对局增长的数据”：

1. 玩家与宠物的关系：`(:Player)-[:HAS]->(:Pet)`
2. 宠物之间的羁绊关系：`(:Pet)-[:HAS_BOND]->(:Pet)`

它们共同服务于：对局结束后“更新羁绊等级/强度”，让后续对战的 AI 能基于历史组合做更好的选择与解释。

## 1) `Player -[:HAS]-> Pet`：玩家拥有哪些宠物、等级如何变化

### 当前实现写入逻辑（关键点）

- 写入触发点：仅在 **battle 结束** 且对应 side 有 `human` 且 `userId` 存在时。
- 写入内容：
  - `petIds` = 战斗结束时该 side 阵容的 3 只宠物
  - 对每个宠物，执行：
    - `MERGE (pl:Player {id: $playerId})-[h:HAS]->(p:Pet)`
    - 当边首次创建：`h.battleCount = 0, h.level = 1`
    - 每场战斗累计：
      - `h.battleCount = h.battleCount + 1`
      - `h.level = floor((battleCount + 1) / 5) + 1`

### 为什么把“玩家-宠物”关系建成边？

- 一对一映射不需要额外中间表：关系边天然适合表达“用户对某资源的拥有/培养状态”。
- 可扩展：未来如果需要记录“该玩家对宠物的熟练度曲线”“使用次数分季节/分模式”，可以在 `HAS` 边上继续加属性，而不改变节点结构。

## 2) `Pet -[:HAS_BOND]-> Pet`：宠物组合历史如何转成羁绊等级

### 当前实现写入逻辑（关键点）

- 写入触发点：同样只在 battle 结束时更新。
- 计算对局发生的宠物对：
  - 对该 side 阵容中的 3 只宠物，枚举两两组合 `(i, j)`，共 3 对。
  - 每个无序宠物对在本场里计数 `battles = 1`
- 图上关系更新：
  - `MERGE (a:Pet)-[r:HAS_BOND]->(b:Pet)`
  - 若边不存在：初始化 `r.battles = 0, r.level = 1`
  - 增量更新：
    - `r.battles = r.battles + pb.battles`
    - `r.level = floor(r.battles / 3) + 1`

### 为什么 Pet-Pet 选“边”（HAS_BOND）而不是节点？

- “羁绊”本质是“两个宠物之间”的属性（pairwise feature），用边最自然。
- 边上挂 `battles/level`，查询时可以直接：
  - `MATCH (a:Pet {id:...})-[r:HAS_BOND]->(b:Pet {id:...})`
  - 无需额外聚合表或冗余节点。

### 无序对的处理方式

代码在写入时把宠物对做了 `sort().join('|')` 的 pairKey 来保证无序一致性；查询时也用了 `UNION` 处理 `a->b` 与 `b->a` 两种方向。

## 3) 关系数据与战斗规则的衔接（为什么这些字段有用）

- 引擎内部羁绊加成/暴击/连携可用条件会依赖“羁绊等级/羁绊曲线”。
- Neo4j 提供的 `HAS_BOND.level` 与 `HAS_BOND.battles` 是长期记忆：
  - AI/解说模块可查询并解释“为何某组合更强/更常触发连携”等。

## 4) 用户操作与对局数据该怎么划分（本模型背后的原则）

当前划分大致是：

1. **短期、强时序的对局过程**（轮次、双方每手动作、事件流）
   - 保存在内存中的 `BattleSession.state/events`
   - 只用于“实时对战决策/合法性校验”
2. **长期、可复用的成长统计**（玩家拥有哪些宠物、宠物对共同上阵次数）
   - 写入 Neo4j 图谱关系：
     - `Player-HAS-Pet`
     - `Pet-HAS_BOND-Pet`
3. **身份与会话认证**
   - 写入 Postgres（`users`, `user_sessions` 等）

为什么这么划分？

- 降低写入量：如果把每个动作/每个事件都存图，数据量与复杂性会爆炸；而我们真正需要用于 AI 决策的是“长期组合统计”，所以只落聚合后的 bond/player 关系。
- 减少耦合：实时对战状态与长期成长更新解耦；图谱不可用时也不应阻塞战斗结果返回（代码在 persist 失败时不阻断）。

优点：

- 更稳：Neo4j 可用性不影响对局完成与返回。
- 更快：查询“组合强弱/可解释性”只需要少量图查询。
- 更易演化：你可以逐步增加图上属性（例如给 `HAS`/`HAS_BOND` 加时间窗、模式维度），但不影响实时引擎。
