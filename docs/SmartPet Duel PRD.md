# SmartPet Duel智宠对决 PRD 正式版v2.3

**项目定位**：基于知识图谱（Neo4j）+ 多智能体（LangGraph）+ 可选向量增强（GraphRAG Phase 2）的原创文字策略对战游戏

**文档状态**：已锁定 | **版权说明**：全原创无IP风险 | **适用场景**：本地开发、项目管理、Cursor同步开发

---

## 一、项目基础信息

### 1. 项目名称

中文名：智宠对决

英文名：SmartPet Duel

### 2. 核心定位

轻量化、高策略、图谱驱动型宠物回合对战游戏，摒弃传统抽卡、数值碾压玩法，以**动态宠物羁绊网络、LangGraph 对战 AI（含规则 fallback）、Neo4j 图谱驱动成长**为核心亮点；**自然语言战术查询与向量增强为 Phase 2 可选**（见第八节、附录B）。技术栈覆盖 LangChain、LangGraph 与可选 GraphRAG，适合技术演示与本地项目开发。

### 3. 核心技术栈

- 后端框架：Hono（Node.js新一代高性能框架）

- 知识图谱：Neo4j（MVP：Cypher/模板查询；Phase 2 可选：向量 + GraphRAG 增强查询）

- AI模块：LangChain + LangGraph（对战 AI；可选解说员 / 战术 RAG，见第十节、附录B）

- 通信模式：等待式API（无轮询、低带宽、无服务器压力）

- 前端呈现：纯文字固定面板UI

- 数据存储：本地JSON配置 + Neo4j图谱数据库

### 4. 核心游戏模式

3v3轮换1v1回合制对战，双方各携带3只宠物，单只宠物上场对决，阵亡后自动轮换，率先击溃对方全部宠物即为获胜。

---

## 二、核心玩法流程

1. 玩家/AI组建3只宠物的出战阵容，确认后开启对战

2. 系统随机/先手判定决定首回合出手顺序，同一时间仅1只宠物在场对决

3. 每回合执行操作：普通技能释放、宠物切换、合体技能释放（满足条件）

4. 上场宠物血量清零则判定阵亡，自动切换阵容内下一只宠物上场

5. 循环对战，直至一方3只宠物全部阵亡，另一方获得对战胜利

6. 对战结束后，自动更新宠物羁绊、等级数据，同步至知识图谱

7. **（若启用）** 服务端生成 **战斗回放** 事件流归档；**PvP** 对局按 **第十节** 更新 **天梯** 积分与榜单

## 四、属性克制规则（平衡定稿·无BUG）

火 → 草：1.5倍伤害

草 → 水：1.5倍伤害

水 → 火：1.5倍伤害

电 → 水：2.0倍伤害

光 → 电、灵：1.2倍伤害

灵 → 光、电：1.2倍伤害

光/灵对火、水、草属性：1.0倍普通伤害，无额外加成

## 五、等级与成长系统（图谱驱动·无经验）

### 1. 宠物等级成长规则

全程无打怪、无经验值，仅依靠宠物共同出战次数升级，共3个等级：

- Lv.1 → Lv.2：单只宠物与任意宠物共同出战满10场

- Lv.2 → Lv.3：单只宠物与任意宠物再次共同出战满20场

- 满级累计：共同出战满30场

### 2. 等级属性加成

- Lv.1：初始基础HP、基础攻击

- Lv.2：HP提升50%，攻击提升50%

- Lv.3：HP提升100%，攻击提升100%

### 3. 宠物羁绊等级（双宠之间）

羁绊为两只宠物间的专属关系，依靠共同出战次数提升，共3级：

- Lv.1（共战5场）：暴击率+5%

- Lv.2（共战10场）：技能伤害+10%

- Lv.3（共战20场）：解锁专属合体技能

---

## 六、技能体系规则

### 1. 基础技能规则

每只宠物固定拥有2个普通技能，可在回合内正常释放，无前置解锁条件。

### 2. 合体技能规则（核心）

- 解锁条件：双宠羁绊等级达到Lv.3

- 释放限制：合体搭档宠物必须存活且在出战阵容内，搭档阵亡则合体技立即锁定，无法释放

- 冷却规则：释放后冷却2回合，期间不可再次使用

- 每场对战，单组合体技仅限使用2次

### 3. AOE技能规则

- 对当前场上敌方宠物造成100%全额伤害

- 对敌方替补未上场宠物造成30%溅射伤害

- 替补宠物无法被AOE技能直接秒杀，溅射伤害仅用于压低血线

## 七、宠物获取机制（无抽卡·纯羁绊解锁）

摒弃商城、抽卡、付费解锁模式，纯依靠知识图谱关系、对战行为解锁宠物：

1. **羁绊解锁**：现有宠物羁绊等级达标，自动解锁关联宠物

2. **战术成就解锁**：完成指定对战操作、达成战术目标，解锁稀有宠物

3. **图谱探索解锁**：通过GraphRAG自然语言查询，获取隐藏宠物线索，完成条件解锁

4. **声望解锁**：累计对战场次、胜率达标，解锁高阶特殊属性宠物

**数据定稿**：具体解锁条件、成就 ID、图谱线索与服务器判定规则见 **附录A：解锁条件矩阵**；隐藏宠物须 **服务端校验条件** + **线索碎片/图谱节点**，避免仅靠自然语言穷举剧透。

## 八、宠物关系图谱（GraphRAG核心）

### 1. 图谱节点类型

- Player：玩家节点，记录拥有宠物、对战数据

- Pet：宠物节点，记录属性、等级、基础数值

- Skill：技能节点，关联宠物技能、合体技

- Attribute：属性节点，承载克制关系

- **（扩展，与第十节对齐）** `Battle` / `Match` 会话节点（可选建模）：对局 ID、`ruleset_id`、胜负、是否计入天梯；或等价信息存关系型/文档存储并由图谱边 `[:PLAYED]->` 关联至 Player

### 2. 核心关系链路

- 玩家 -[:HAS]-> 宠物：玩家拥有宠物的归属关系

- 宠物 -[:BATTLE_WITH]-> 宠物：双宠共同出战记录

- 宠物 -[:HAS_BOND]-> 宠物：动态羁绊等级关系

- 宠物 -[:UNLOCK_BY]-> 宠物：宠物解锁关联关系

- 属性 -[:COUNTER]-> 属性：属性克制关系

### 3. 图谱核心作用

- 实时计算、更新宠物羁绊与等级数据

- 支撑战术与阵容查询（MVP 以 Cypher/结构化查询为主；自然语言与向量检索见下节分层）

- 为对战 AI（LangGraph）提供结构化策略依据（克制、羁绊、合法动作集合）

- 自动判定宠物解锁条件

### 4. GraphRAG与查询能力分层（MVP / Phase 2）

- **MVP（必选）**：Neo4j 存储实体与关系；服务端通过 **Cypher** 或预编译查询模板完成羁绊、解锁、克制、阵容合法性校验；保证低延迟、可测试、无 LLM 依赖路径。

- **Phase 2（可选）**：在 MVP 稳定后引入 **向量检索**（历史战报/阵容片段 embedding）与/或 **NL→Cypher（严格 allowlist）**，用于自然语言战术查询、相似阵容建议；**不改变战斗数值**，查询结果仅作展示或建议层。

- **原则**：全链路「为 GraphRAG 而 GraphRAG」不纳入 MVP 验收；Phase 2 可标为演示/增强模式单独开关。

## 九、完整对战规则

1. 轮换规则：3v3阵容，单次仅1只宠物上场对决，阵亡后自动轮换下一只

2. 回合流程：双方操作执行 → 伤害结算 → 阵亡判定 → 宠物轮换 → 回合结束

3. 伤害计算公式：最终伤害 = 宠物攻击 × 技能系数 × 属性克制系数 × 羁绊加成

4. 阵亡规则：宠物本场阵亡不可复活，不影响场外羁绊、等级数据

5. 胜利条件：击溃对方全部3只出战宠物

6. 超时规则：玩家超时未操作，自动切换AI托管执行操作

7. 断线重连：支持断线后重连，恢复当前对战状态（会话持久化、版本号与 TTL 等实现要求见 **十五、非功能需求（NFR）**）

8. **战斗回放**：每场**玩家参与的对战**（人机或人人，以产品实现范围为准）在结束后生成 **回放记录**；玩家可从列表选择回放，由客户端或服务端按 **确定性事件流** 逐步重演（规则与 **十·1**、**十五·6** 一致）。

## 十、战斗回放与天梯排行

### 1. 战斗回放

**目标**：支持查看历史对局的逐步过程，用于复盘、分享与争议核对；**不**改变已结束对局的官方结果。

**记录内容（事件流 / Event Sourcing 思路）**：

- **规则版本号**：`ruleset_id`（与对战引擎逻辑版本绑定，回放时必须一致否则提示不兼容）。

- **随机种子**：开局与关键随机点所用种子（与 NFR 中确定性要求一致）。

- **有序事件列表**：每回合双方 **合法操作** 及引擎产出的 **结算事件**（伤害、阵亡、轮换等），以服务端权威日志为准；**禁止**仅客户端拼装结果冒充回放。

**重演方式**：使用与实时对战 **同一套对战引擎** 重放事件流；重演结果须与当时胜负、最终血量一致（回归测试纳入附录B 相关条目）。

**范围与存储**：回放列表为 **纯文字界面**（场次摘要、时间、结果）；单条回放体积与 **保留天数 / 条数上限** 见 **十五、非功能需求（NFR）**；仅保存 **必要字段**，避免冗余全量聊天或 Phase 2 解说全文（若需可单独开关存储解说文本）。

### 2. 天梯排行

**目标**：在 **玩家对战（PvP）** 场景下提供竞技排序与榜单展示（**人机对战不计入天梯**，除非产品另行开关；默认 PRD：**仅 PvP**）。

**积分模型（可配置，MVP 建议）**：

- 采用 **Elo 或简化 Trueskill 变体** 等成对竞技评分；胜方加分、负方减分，**仅服务端**根据对局结果更新，客户端不可自报分数。

- 可选 **赛季/重置**（本地演示可为单赛季长期榜）。

**榜单维度**：全服或好友范围由实现决定；**纯文字面板** 展示排名、昵称（或匿名 ID）、积分、胜负场次（可选）。

**与图谱**：可将 `Player` 节点扩展属性或边关联 **当前天梯分、最近更新时间**（具体 schema 实现层定稿）；**解锁与声望**（第七节）若引用胜率/场次，须与 **权威对局记录** 一致，避免与天梯数据冲突。

**公平性**：仅统计 **服务端校验结束** 的对局；断线判负、超时托管等规则与 **第九节** 一致，并在对局记录中可追溯。

## 十一、AI智能体系

### 1. 对战引擎（服务端，确定性，非 LLM）

原「裁判 AI」更名为 **对战引擎 / GameEngine**：负责伤害计算、回合顺序、阵亡判定、羁绊与等级数据更新、图谱同步。**全部为确定性规则**，与 LLM 无关；胜负与数值 **仅以服务端计算为权威**，客户端不得信任本地推算结果。

### 2. 对战 AI（决策端，LangGraph）

基于 **LangGraph** 实现 **状态机流转**，从 Neo4j/Cypher 或 Phase 2 的 RAG 层读取 **结构化** 克制、羁绊、合法动作集合；输出本回合操作。**不承诺「博弈论最优」**：验收以 **强度档位**、**合法动作内排序**、**固定随机种子下可复现** 为准（见附录B）。

**多智能体分工（单 pipeline，可共用同一底层模型或纯规则 fallback）**：

| 节点 | 职责 | 输入/输出 |
|------|------|-----------|
| Analyst | 解析战局快照（JSON），拉取克制、羁绊、合体技可用性 | 结构化特征向量 |
| Planner | 枚举合法动作（技能/切换/合体），打分排序 | 带分数的候选动作列表 |
| Persona | 将排序结果按性格偏置（激进/保守/策略）重排或截断 | 最终 1 个动作 |

**性格**：通过 **同一模型的 system prompt 或可调参数** 实现三种风格，**不**要求三个独立大模型。

**失败回退**：LLM 不可用、超时或输出非法动作时，**自动回退规则 AI**（启发式：优先克制、残血收割、合体技条件满足时按档位释放等）。

### 3. 战术 RAG（可选，Phase 2，二选一或组合）

- **方案 A**：Embedding + 向量库，检索相似历史战报/阵容，输出 **文字建议**，**不改变战斗数值**。

- **方案 B**：NL→Cypher 或 NL→**预定义查询模板**（allowlist），仅用于查询与教程；**禁止** LLM 直接写库或直接改写对战状态。

### 4. 解说员（LLM，可选）

基于 **结构化战斗事件**（技能名、伤害数字、阵亡）生成每回合或关键事件 **1～2 句解说**；约束：**不得编造未发生的数值或技能**。可通过配置 **单独开关**，控制成本与延迟。

### 5. 内容侧 Embedding（可选）

宠物/技能短描述向量化，用于「相似宠物」、**阵容推荐解释**（解释性展示，非改数值）。

### 6. 评测与可观测性

- **AI vs AI** 批量对战：统计胜率、平均回合数；**固定种子** 回放用于回归。

- 日志：对战流水、（若启用）LangGraph 决策轨迹、图谱写入失败 **重试** 策略见 NFR。

## 十五、非功能需求（NFR）

### 1. 状态持久化与断线重连

- 每局对战需 **会话 ID**、**状态版本号**（防并发覆盖）、**TTL**（超时清理）。

- 存储介质选型由实现决定（内存/Redis/DB），PRD 要求：**重连后能恢复与等待式 API 一致的当前局面**。

- **并发**：同一对局串行处理指令；冲突时以服务端版本为准。

### 2. LLM 与成本

- API Key **服务端** 保管；环境变量或密钥管理，**不入库**、**不进客户端**。

- **单次对战 token 上限**、**速率限制**；超限或 **无网** 时 **fallback 规则 AI** 与关闭解说。

- 模型选型记录于开发配置（可变更），PRD 不绑定具体厂商。

### 3. 安全与查询

- 自然语言/LLM 查询：**仅返回 allowlist 字段**；**禁止** 越权读取其他玩家数据。

- **提示注入**：NL→查询链路须模板化或二次校验，**禁止** 任意 Cypher 拼接。

- 客户端仅提交 **意图 ID + 参数**；敏感逻辑只在服务端执行。

### 4. 可观测性与可靠性

- 结构化对战日志；可选 LangGraph **决策轨迹** 日志（调试开关）。

- 图谱写入 **失败重试**（次数/退避）与最终一致性说明（例如战后重放补偿）。

### 5. 测试与验收

- **确定性回放**：相同种子 + 相同输入 → 相同结果（引擎层）。

- **AI vs AI** 冒烟：至少跑通多局并产出胜率/回合统计（阈值见附录B）。

### 6. 战斗回放与存储

- 单用户 **回放条数上限**、**单条大小上限**、**保留 TTL**（如 30 天可配置）需在环境配置中声明。

- **旧版 `ruleset_id` 回放**：须明确提示无法重演或仅展示摘要，避免静默错误。

- **隐私**：回放元数据是否展示对手完整昵称由产品配置；榜单与对外展示遵循最小必要原则。

### 7. 天梯

- **分数更新**与对局结束 **同一事务或最终一致**（失败重试、防双计分）。

- **防刷**：短时异常胜负模式可触发风控标记（实现可选，PRD 要求至少 **服务端唯一计分入口**）。

## 附录B：LangGraph 与 AI 模块验收标准

### B.1 LangGraph 状态机（要求）

- 须实现 **Analyst → Planner → Persona** 管线；任一节点失败或超时 → **回退规则 AI**，并记录日志。

- 输出须为 **合法动作**（在服务端枚举内）；非法输出丢弃并重试或回退。

### B.2 对战 AI 强度与复现

- **固定随机种子** 下，相同局面输入 → **相同输出动作**（在启用 LLM 时若无法保证，则验收 **规则 AI 路径** 可复现，LLM 路径标注为概率性）。

- **强度档位**：至少 **易 / 中 / 难** 三档（映射 Persona + Planner 权重或规则深度）；**难档对中档** 应有显著胜率优势（具体百分比由测试在阶段六根据平衡微调，PRD 只要求 **可测量**）。

### B.3 战术 RAG / NL 查询（Phase 2）

- 验收：**仅返回 allowlist 内数据**；**无** 越权；**无** 任意图数据库写操作。

### B.4 解说员（可选）

- 验收：解说文本 **不编造** 未在事件流中出现的数值；开关关闭时零额外延迟。

### B.5 AI vs AI 评测

- 自动化跑 **N 局**（N≥100 为建议值），输出 CSV 或日志：**胜率、平均回合数**；作为回归基线。

### B.6 战斗回放回归

- 抽样保存 **事件流 + ruleset_id + 种子**，自动化 **重演 N 条**，断言胜负与关键结算与归档一致。

### B.7 天梯数据一致性

- 抽样对局：对局结果 → 积分变更 → 榜单读回，**链式校验**无丢失、无重复计分。
